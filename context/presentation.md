# Presentation Strategy

## Project Ordering

Lead with strongest signal, end with breadth.

### Tier 1 — "I build real things"
1. **Zwipe** - largest project, shows full-stack architecture at scale
2. **Halo Action Importer** - production tool, shows resilience engineering
3. **Halo Custom Field Builder** - shipped tool, shows end-to-end delivery

### Tier 2 — "I understand the ecosystem"
4. **Marvin** - AI agent tooling in Rust (tool use, streaming, context management)

### Tier 3 — "I go deep on things I don't know"
5. **Nighthawk** - database internals from scratch
6. **Upsee** - ML inference pipeline on-device
7. **Capture** - cross-platform systems programming

## Per-Project Page Structure

Each project page has:

1. **Header** - name, headline, category tag, GitHub link
2. **Objective** - what it does and why, 2-3 sentences
3. **Approach** - bullet points on architecture and design decisions
4. **Implementation** - curated code snippets showing the interesting technical decisions
5. **Obstacles** - honest, specific challenges encountered and how they were solved
6. **Progress & Impact** - current state and what it demonstrates

## Home Page

- Brief intro: "I build production Rust systems that solve hard problems."
- Project cards with impact metrics and hover tooltips for detailed context
- No "about me" essay. Let the projects speak.

## Impact Tooltips

Each project card has a short impact metric that shows a detailed tooltip on hover:
- **Zwipe**: "24,500 lines of production Rust" - details about hexagonal architecture, type safety, extensibility
- **Halo Action Importer**: "Weeks of manual work, automated" - explains Halo's browser import limitations vs this tool
- **Halo Custom Field Builder**: "Manual UI clicks to one CSV import" - describes the CSV workflow and speed gains

## Tone

- Direct. No corporate speak.
- Honest about what's a learning project vs what's production code
- Technical but accessible
- No mention of employer. Focus on what was built and why it matters.
- Avoid em dashes. Keep punctuation clean.

## Halo References

- Refer to "Halo Software" or "Halo Software products" (not just ITSM)
- No parenthetical product lists (ITSM, PSA, CRM...)
- The tools work across the Halo suite. Keep it simple.

## What NOT to Include

- No "skills" section with progress bars
- No star ratings on technologies
- No timeline/resume format
- No blog posts unless they add real value
- No description of employer/role. This is a project showcase.
